# PRD: АСУ «Техника» — система учёта и контроля техники компании

**Версия документа:** 1.0
**Дата:** 03.07.2026
**Статус:** MVP реализован (прототип на статичном HTML/JS)
**Владелец продукта:** ИТ-отдел

---

## 1. Обзор и цель проекта

Компании требуется единая система для учёта, контроля и анализа парка техники: от момента поступления на склад до выдачи сотрудникам, ремонта, возврата и списания. Сегодня учёт ведётся разрозненно (таблицы, бумажные акты), из-за чего сложно быстро ответить на вопросы: сколько техники на складе, у кого что находится, кто материально ответственен, какова история конкретной единицы.

**Цель продукта** — дать материально ответственному лицу (МОЛ) из ИТ-отдела инструмент для оперативного учёта техники и приёма/передачи, а руководству компании — готовую аналитику и отчётность без ручного сведения данных.

### 1.1 Задачи продукта
- Централизованный реестр техники с моделью, серийным номером, статусом.
- Учёт приёма на склад и выдачи со склада с фиксацией ответственных лиц.
- Полная история перемещений каждой единицы техники (аудиторский след).
- Наглядная аналитика и дашборды для руководства (без запроса отчётов у ИТ вручную).
- Быстрый поиск и иерархическая навигация по категориям техники.

### 1.2 Метрики успеха
| Метрика | Целевое значение |
|---|---|
| Время на оформление приёма/выдачи техники | ≤ 1 минуты |
| Время подготовки отчёта для руководства | Мгновенно (вместо ручного свода за 1–2 дня) |
| Доля техники с неизвестным местонахождением/ответственным | 0% |
| Время на поиск истории конкретной единицы техники | ≤ 10 секунд |

## 2. Пользователи и роли

| Роль | Кто | Основные задачи в системе |
|---|---|---|
| **МОЛ (материально ответственное лицо)** | Сотрудник ИТ-отдела | Ведёт реестр, оформляет приём/выдачу, следит за статусами, отвечает за сохранность техники |
| **Руководство компании** | Директора, руководители подразделений | Просматривает дашборды, аналитику, отчёты; принимает решения по закупкам/списанию |
| *(Будущая роль)* Сотрудник-пользователь техники | Любой сотрудник компании | Просматривает список закреплённой за ним техники (не входит в MVP) |

На MVP-этапе разграничение прав доступа между ролями не реализовано — все функции доступны единому пользователю интерфейса. Ролевая модель — см. раздел 9 «Дальнейшее развитие».

---

## 3. Объём и формат решения

- **Тип:** одностраничное веб-приложение (SPA), single HTML-файл.
- **Платформа:** браузер, адаптивная вёрстка (десктоп + мобильные устройства).
- **Хранение данных (MVP):** локально в браузере (localStorage), с демо-данными для приёмки.
- **Доступ:** без авторизации на этапе MVP (см. раздел 9 — интеграция с реальной БД и ролями вынесена в бэклог).

## 4. Структура и навигация

Главный экран построен вокруг **иерархического дерева разделов** в левой панели, раскрывающегося по клику:

```
├── Дашборд
├── Реестр техники
│   ├── Ноутбуки
│   ├── Стационарные ПК
│   ├── Мониторы
│   ├── Принтеры и МФУ
│   ├── Мобильные телефоны
│   ├── Сетевое оборудование
│   ├── Периферия
│   └── Серверное оборудование
├── Склад
│   ├── Приём на склад
│   └── Выдача со склада
├── История передвижений
├── Пользователи и МОЛ
│   ├── Пользователи техники
│   └── Материально ответственные лица
└── Аналитика и отчёты
```

Каждый узел дерева показывает счётчик единиц техники. Реализован поиск по дереву. На мобильных устройствах дерево скрывается в выдвижную панель по кнопке-гамбургеру.

## 5. Функциональные требования

### 5.1 Дашборд (главная страница)
- KPI-плашки: всего техники, на складе, выдано, в ремонте, списано (с долей от общего парка).
- График «Движение техники по месяцам» — приём и выдача за 12 месяцев (столбчатая диаграмма).
- Диаграмма распределения техники по категориям (кольцевая).
- Диаграмма по статусам техники (горизонтальные столбцы).
- Диаграмма по подразделениям — сколько техники закреплено за отделом.
- Лента из 10 последних операций (приём/выдача/возврат/ремонт/списание).

### 5.2 Реестр техники
- Просмотр всей техники в иерархическом виде по категориям (аккордеон, раскрывается/сворачивается).
- Для каждой единицы техники отображается: наименование/модель, серийный номер, инвентарный номер, статус, текущий пользователь, МОЛ, подразделение.
- Фильтрация: по статусу, по МОЛ, полнотекстовый поиск (модель, серийный номер, инв. номер, пользователь).
- Отдельная страница по каждой категории с той же логикой фильтрации в табличном виде.
- Карточка техники (модальное окно) с полной информацией и **историей перемещений в виде таймлайна** (все операции по единице техники в хронологическом порядке).
- Списание техники прямо из карточки (с подтверждением и фиксацией в истории).
- Добавление новой техники вручную (категория, модель, серийный номер, инвентарный номер, стоимость, дата приёма, ответственное МОЛ).

### 5.3 Приём техники на склад
Поддерживаются три сценария:
1. **Новое поступление** (закупка) — ввод модели, серийного номера, категории, МОЛ, даты, комментария.
2. **Возврат от пользователя** — выбор техники из числа выданной, автоматическое обновление статуса на «На складе».
3. **Возврат из ремонта** — аналогично, для техники в статусе «В ремонте».

Каждая операция создаёт запись в истории передвижений. Отображается список последних приёмов.

### 5.4 Выдача техники со склада
- Форма выбора: техника (только со статусом «На складе»), сотрудник-получатель, МОЛ, дата, комментарий.
- Подразделение получателя подставляется автоматически.
- После выдачи статус техники меняется на «Выдана», фиксируется движение в истории.
- Список свободной техники на складе, доступной к выдаче.

### 5.5 История передвижений
- Полный журнал всех операций: приём, выдача, возврат, передача в ремонт, списание.
- Для каждой записи: дата, тип операции, техника (модель + серийный номер), от кого/кому, ответственный МОЛ, комментарий.
- Фильтры: по типу операции, по диапазону дат, полнотекстовый поиск.
- Постраничная навигация.

### 5.6 Пользователи и МОЛ
- **Пользователи техники**: список сотрудников с подразделением, должностью, количеством закреплённой техники, отметкой «МОЛ».
- **МОЛ**: отдельный раздел — сколько единиц техники закреплено за МОЛ, сколько на складе, сколько выдано под его ответственностью.
- Возможность добавить нового пользователя или назначить сотрудника материально ответственным лицом.

### 5.7 Аналитика и отчёты (для руководства)
- KPI: балансовая стоимость парка техники, средний возраст техники, общее количество операций, число назначенных МОЛ.
- График стоимости парка по категориям.
- Распределение техники по возрасту эксплуатации (до 1 года / 1–2 / 2–3 / более 3 лет).
- Количество операций по типам (приём/выдача/возврат/ремонт/списание).
- Топ-6 подразделений по количеству закреплённой техники.
- Выгрузка сводного отчёта по движению техники в CSV.
- Выгрузка полного реестра техники в CSV (доступно из любого раздела через кнопку в шапке).

## 6. Модель данных

### 6.1 Equipment (Техника)
| Поле | Тип | Описание |
|---|---|---|
| id | string | Уникальный идентификатор |
| category | enum | Категория (ноутбук, ПК, монитор, принтер, телефон, сеть, периферия, сервер) |
| name | string | Модель/наименование |
| serial | string | Серийный номер |
| inv | string | Инвентарный номер |
| cost | number | Балансовая стоимость |
| purchaseDate | date | Дата приёма на баланс/склад |
| status | enum | На складе / Выдана / В ремонте / Списана |
| userId | ref → User | Текущий пользователь (если выдана) |
| molId | ref → User | Ответственное МОЛ |

### 6.2 User (Пользователь / МОЛ)
| Поле | Тип | Описание |
|---|---|---|
| id | string | Уникальный идентификатор |
| name | string | ФИО |
| dept | string | Подразделение |
| position | string | Должность |
| isMol | boolean | Признак материально ответственного лица |

### 6.3 Movement (Движение / история)
| Поле | Тип | Описание |
|---|---|---|
| id | string | Уникальный идентификатор |
| equipId | ref → Equipment | Единица техники |
| type | enum | Приём / Выдача / Возврат / Передача в ремонт / Списание |
| date | date | Дата операции |
| from / to | string | Источник и получатель (склад, сотрудник, сервис, поставщик) |
| molId | ref → User | Ответственное МОЛ на момент операции |
| userId | ref → User | Пользователь (если применимо) |
| comment | string | Комментарий/основание операции |

## 7. Нефункциональные требования

- **Адаптивность:** корректное отображение на экранах от 360 px (мобильные) до широких десктопных мониторов.
- **Производительность:** отклик интерфейса на фильтрацию/поиск — мгновенный (без перезагрузки страницы).
- **Целостность данных:** каждая смена статуса техники обязательно сопровождается записью в историю передвижений — не должно быть операций «в обход» журнала.
- **Доступность интерфейса:** видимый фокус для клавиатурной навигации, достаточный контраст текста.
- **Локализация:** интерфейс на русском языке, формат дат ДД.ММ.ГГГГ, валюта — тенге (₸).

## 8. Технический стек (текущая реализация)

- HTML5 / CSS3 (адаптивная сетка, CSS-переменные для темизации).
- Vanilla JavaScript (без фреймворков) — управление состоянием, рендеринг разделов.
- Chart.js — визуализация графиков и диаграмм.
- Хранение данных — localStorage браузера (временное решение для MVP/демо).

> Ограничение текущей реализации: данные хранятся только в браузере конкретного устройства и не синхронизируются между пользователями. Это осознанное упрощение MVP, требующее замены на серверное решение при переходе в продуктив (см. раздел 9).

## 9. Дальнейшее развитие (вне рамок MVP)

1. **Серверное хранилище и API** — перенос данных из localStorage в базу данных (PostgreSQL/MySQL) с REST/GraphQL API, чтобы данные были едиными для всех пользователей.
2. **Авторизация и ролевая модель**:
   - МОЛ — полный доступ к вводу/редактированию;
   - Руководство — доступ только на просмотр дашбордов и отчётов;
   - Сотрудник — просмотр только своей закреплённой техники.
3. **Печатные формы** — генерация актов приёма-передачи техники (PDF) с подписями.
4. **Уведомления** — напоминания о сроках гарантии, плановом обслуживании, долгом нахождении техники в ремонте.
5. **Штрихкоды/QR-коды** — печать и сканирование меток для быстрой инвентаризации.
6. **Импорт данных** — загрузка существующего реестра техники из Excel/1С при внедрении.
7. **Журнал аудита действий пользователей** (кто и когда внёс изменения в систему).
8. **Мобильное приложение или PWA** — для инвентаризации техники непосредственно на месте с камерой телефона.

## 10. Вне рамок текущей версии (Out of Scope)

- Многопользовательская синхронизация в реальном времени.
